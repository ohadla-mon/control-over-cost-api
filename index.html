Contract Against Product Metering

We consume product metering's data — we don't dictate how it's stored, only that the read contract supports filtering, grouping, and sorting on every first-class dimension. Mechanics stay metering's call.

7.1  Supported Filters

The current page filters by user only — feedback was that this is far too thin. Each filter below must compose with AND, be deep-linkable, and be saveable as a set. Every one maps to a first-class field so it stays fast at account scale.

FilterMaps toWhy we need it
Billing cyclebilling_cycleScope every view to the right cycle
Date rangedate_from · date_toAn explicit window, independent of billing cycle
FeaturefeatureThe pie; one credit language
User (who's charged)user_idToday's single billing field
Entity (type + id)entity_type · entity_idThe feature's main object — a board, workflow, automation, agent, item…
Trigger (type + id)trigger_type · trigger_idWhat caused it — a user, workflow, board, agent…
Over-limit users onlytotal vs. limitGovernance triage

Dimension filters  — Everything else lives in each feature's dimensions (§05) rather than the core record, so this list is illustrative, not exhaustive — it grows as features add dimensions. Same rule: filterable and groupable only once first-class.

FilterMaps toWhy we need it
Agentagent_idDrill into a specific agent's runs
Appapp_idDrill into a specific app's usage
Columncolumn_idDrill into a specific AI column
Column typecolumn_typeWhich AI operation — translate, summarize, generate…
Object / item / meetingobject_idTrace to the concrete object

Search is a single typeahead that resolves names to IDs (board, workflow, agent, user, item, meeting) and applies the matching ID filter — so an admin can type a board name and land on that board's spend without knowing its ID.

7.2  Aggregation, Grouping & Pagination

Every request to the contract is the same pipeline — filter → group → sort → paginate — and the four axes below compose freely. How payments computes the result (pre-aggregated rollups, on-the-fly, cache) is its call; the requirement is what each axis supports, plus speed at account scale.

Filter

Every filter from §7.1 — core or dimension — AND'd together and applied first, so an aggregate only ever sums the rows a filtered view would show.

Group by

Zero or more dimensions, combined freely: feature, user_id, entity_id, trigger_type, or a date bucket (day / week / month). Zero dimensions returns one grand-total row.

Sort by

Any dimension in the current group-by, or the metric itself (credits, count) — ascending or descending. Changing the group-by or sort resets the pagination cursor.

Paginate

A stable keyset cursor over the grouped, sorted result. Accounts run to tens of thousands of rows — no deep-page slowdown.

ℹ️

Refunds net correctly. A refund is a negative credits event and must net into every total, at every grain — so a corrected charge shows the true figure everywhere without special-casing on the surface.

7.3  Read API Endpoint Reference

Three endpoints, not five — /credits/query is one flexible aggregate endpoint parameterized by group_by, so the pie, the per-user list, the per-user-per-feature split, each feature's entity table, top owners, and the time-series trend are all the same endpoint with a different filter/group-by/sort combination. All three share account-scoped auth, rate limits, and the full §7.1 filter grammar.

GET /credits/account

The account's entitlement snapshot — total, used, and left for the cycle. The source for "credits remaining" in §4.2.

Query Parameters

POST /credits/query

One flexible aggregate endpoint — any combination of filters + group-by + sort. This is the pie, the per-user list, the per-user-per-feature split, each feature's entity table, top owners, and the time-series trend — same endpoint, different group_by. POST because the body carries arrays, not just scalar params.

Request Body

FieldDescription
filtersObject — any combination of §7.1 fields (feature, user_id, board_id, workflow_id, automation_id, agent_id, app_id, column_id, object_id, trigger_type, trigger_id, over_limit), AND'd together. billing_cycle or date_from/date_to scopes the window.
group_byArray, zero or more of: feature, user_id, entity_id, trigger_type, date:day | date:week | date:month. Empty array returns one grand-total row.
sortArray of {field, order} — field must be a requested group_by dimension or credits/count; order = asc|desc. Multiple entries compose in order given.
cursor, limitKeyset pagination over the grouped, sorted result. Changing group_by or sort resets the cursor.
Example — top owners for one feature
{
  "filters":  { "feature": "ai_workflows", "billing_cycle": "2026-07" },
  "group_by": ["user_id"],
  "sort":     [{ "field": "credits", "order": "desc" }]
}
AxisBehaviour
Groups byFully client-specified via group_by — any dimension, any combination
Sorts byFully client-specified via sort — any grouped field or the metric
Filters byEvery §7.1 filter, freely combined
GET /credits/events

Event-level raw rows behind one drawer — individual runs, items, steps, meetings, or endpoint calls. Not an aggregate, so it stays separate from /credits/query.

7.4  GraphQL Schema Shape

One schema-first /graphql endpoint — the external, self-serve API a customer calls directly against their own account. It mirrors /credits/query's filter / group-by / sort grammar field-for-field; the one deliberate difference is pagination, exposed as Relay-style cursor connections instead of the REST cursor/limit pair.

POST /graphql

Single endpoint, schema-first. The API token is account-scoped, so there's no account_id argument anywhere in the schema — a customer can only ever see their own consumption.

Schema (excerpt)
# one field per REST endpoint in §7.3
type Query {
  creditsAccount(billingCycle: String, dateFrom: Date, dateTo: Date): CreditsAccount!
  creditsQuery(
    filter:  CreditsFilter
    groupBy: [CreditsDimension!]
    sort:    [CreditsSortInput!]
    first: Int = 50, after: String
  ): CreditsQueryResult!
  creditsEvents(
    feature: String!, entityId: ID!
    first: Int = 50, after: String
  ): CreditsEventConnection!
}

input CreditsFilter {   # every §7.1 filter, AND'd together
  feature:       [String!]
  userId:        [ID!]
  boardId:       [ID!]
  workflowId:    [ID!]
  automationId:  [ID!]
  agentId:       [ID!]
  appId:         [ID!]
  columnId:      [ID!]
  objectId:      [ID!]
  triggerType:   [String!]
  triggerId:     [ID!]
  overLimit:     Boolean
  billingCycle:  String
  dateFrom:      Date
  dateTo:        Date
}

enum CreditsDimension {
  FEATURE  USER_ID  ENTITY_ID  TRIGGER_TYPE
  DATE_DAY  DATE_WEEK  DATE_MONTH
}
input CreditsSortInput { field: String!  order: SortOrder = DESC }

type CreditsQueryResult {
  rows:        [CreditsGroupRow!]!
  totalCount:  Int!
  lastUpdated: DateTime!  # §08 "last updated" requirement
}
type CreditsGroupRow {
  dimensions: [DimensionValue!]!
  credits:    Float!
  count:      Int!
}
Example query — top owners for one feature
{
  creditsQuery(
    filter:  { feature: ["ai_workflows"], billingCycle: "2026-07" }
    groupBy: [USER_ID]
    sort:    [{ field: "credits", order: DESC }]
    first: 20
  ) {
    rows { dimensions { key value } credits }
    totalCount
    lastUpdated
  }
}
Response
{
  "rows": [
    { "dimensions": [{ "key": "user_id", "value": "u_8831" }], "credits": 4120 },
    …
  ],
  "totalCount": 340,
  "lastUpdated": "2026-07-05T09:14:00Z"
}
AxisBehaviour
Groups bySame CreditsDimension enum as REST's group_by — one contract, two transports
Sorts byAny requested dimension or the credits/count metric, same as REST
Paginates byfirst/after + pageInfo — Relay connections, not cursor/limit
